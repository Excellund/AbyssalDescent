# Supabase Telemetry Spike Setup

This spike validates server support only.
It does not enable production telemetry upload yet.

## 1) Create a Supabase project

1. Create a free project at https://supabase.com.
2. Open SQL Editor.
3. Run this SQL:

```sql
create table if not exists public.telemetry_spike_events (
  id bigint generated by default as identity primary key,
  created_at timestamptz not null default now(),
  probe_type text not null,
  client_unix bigint not null,
  game_version text not null,
  platform text not null,
  debug_build boolean not null,
  run_summary jsonb not null
);

alter table public.telemetry_spike_events enable row level security;

create policy "spike_insert_anon"
on public.telemetry_spike_events
for insert
to anon
with check (probe_type = 'telemetry_server_support');
```

## 2) Get endpoint + anon key

1. In Supabase project settings, copy Project URL.
2. In API settings, copy the anon public key.
3. Build endpoint:

```
https://<PROJECT_REF>.supabase.co/rest/v1/telemetry_spike_events
```

## 3) Configure game debug settings

Open the Main scene DebugSettings node and set:

- telemetry_spike_enabled = true
- telemetry_spike_endpoint = the endpoint above
- telemetry_spike_api_key = anon key
- telemetry_spike_timeout_seconds = 8.0

Also ensure DebugSettings enabled = true so the spike is allowed.

## 4) Run and validate

1. Run in editor.
2. Check Output log for either:
   - "Telemetry spike probe succeeded (HTTP 201)."
   - or a failure log with result code and HTTP code.
3. In Supabase Table Editor, confirm a new row in telemetry_spike_events.
4. Export Windows build and repeat to confirm runtime parity.

## 5) Expected payload shape

The spike sends:

```json
{
  "probe_type": "telemetry_server_support",
  "client_unix": 1714450000,
  "game_version": "dev",
  "platform": "Windows",
  "debug_build": true,
  "run_summary": {
    "max_depth": 0,
    "rooms_cleared": 0,
    "outcome": "probe"
  }
}
```

## 6) Common failures

- HTTP 401 or 403: incorrect anon key or policy.
- HTTP 404: endpoint path typo.
- HTTP 0 with request_start_failed: URL malformed or DNS/network issue.
- Timeout result: endpoint unreachable from current network.

## 7) Cleanup after spike

When this passes, keep the table for audit and remove permissive spike-only policy before production telemetry rollout.

## 8) Next step after successful spike

Your spike is now validated if you received HTTP 201 and the row appears in Supabase.

Move to hardened server setup:

1. Run SQL from [playtester_telemetry/supabase_production_setup.sql](playtester_telemetry/supabase_production_setup.sql).
This file is also safe to rerun later as a migration when telemetry columns change.
2. Keep the spike table for diagnostics, but route real telemetry to `public.telemetry_runs`.
3. Keep using anon key only in client builds. Never ship a service-role key.
4. Keep RLS enabled and only allow insert checks needed by the client payload shape.

Once this is done, client implementation can proceed safely with:

- run-finish enqueue
- retry queue
- consent toggle and prompt

## 9) Production client config

Set these Project Settings values (or edit `project.godot`) so background uploads can run:

- `application/config/telemetry_upload_endpoint`
- `application/config/telemetry_upload_api_key`

Endpoint format:

```
https://<PROJECT_REF>.supabase.co/rest/v1/telemetry_runs
```

Use the Supabase anon key for `telemetry_upload_api_key`.
